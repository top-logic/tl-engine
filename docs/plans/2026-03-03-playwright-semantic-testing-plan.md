# Semantic Playwright Testing for TopLogic

**Goal:** Establish a Playwright-based E2E testing approach for TopLogic that references **model concepts** (types, attributes) instead of brittle DOM selectors, enabling tests that survive UI evolution with minimal maintenance.

**Status:** Draft / For later consideration

---

## Problem

Classic E2E tests break when the UI changes:

```typescript
// Brittle: breaks on field rename, layout change, CSS refactor
await page.fill('#field_customerName', 'John');
```

TopLogic's model-driven UI is an advantage here: the UI is generated from a typed model, so tests can target model identifiers instead of DOM details.

## Approach (4 Layers)

### Layer 1: Emit `data-testid` attributes in the rendering layer

TopLogic's form rendering generates HTML for model attributes. The rendering layer should emit stable `data-testid` attributes keyed to the model path:

```html
<!-- Generated by TopLogic for attribute "name" on type "my.module:Customer" -->
<input data-testid="my.module:Customer#name" ... >
```

**Where to implement:** The form field rendering code that translates model attributes to HTML input elements. Each `FormField` / `FormMember` that corresponds to a model attribute gets a `data-testid` derived from its model reference (`module:Type#attribute`).

**Scope:** Form fields, table columns, buttons tied to model operations.

### Layer 2: `TLTestHelper` — Playwright helper library

A thin TypeScript library that resolves model references to Playwright locators:

```typescript
// tl-test-helper.ts
import { Page, Locator } from '@playwright/test';

export class TLTestHelper {
    constructor(private page: Page) {}

    /** Locate a form field by model attribute reference. */
    field(modelRef: string): Locator {
        return this.page.locator(`[data-testid="${modelRef}"]`);
    }

    /** Set a form field value by model attribute reference. */
    async setAttribute(modelRef: string, value: string): Promise<void> {
        await this.field(modelRef).fill(value);
    }

    /** Assert a form field displays the expected value. */
    async expectAttribute(modelRef: string, expected: string): Promise<void> {
        await expect(this.field(modelRef)).toHaveValue(expected);
    }

    /** Click a named action/button. */
    async executeAction(actionRef: string): Promise<void> {
        await this.page.locator(`[data-testid="${actionRef}"]`).click();
    }

    /** Navigate to a TopLogic view by its layout component path. */
    async navigateTo(viewPath: string): Promise<void> {
        // Implementation depends on TopLogic's navigation mechanism
        // Could use URL parameters, sidebar clicks, or menu navigation
    }
}
```

### Layer 3: Write tests at the model/semantic level

Tests reference model concepts, not DOM structure:

```typescript
import { test, expect } from '@playwright/test';
import { TLTestHelper } from './tl-test-helper';

test('create a customer with name and email', async ({ page }) => {
    const tl = new TLTestHelper(page);

    await tl.navigateTo('admin/customers');
    await tl.executeAction('create');
    await tl.setAttribute('my.module:Customer#name', 'Acme Corp');
    await tl.setAttribute('my.module:Customer#email', 'info@acme.com');
    await tl.executeAction('save');

    // Verify persistence
    await tl.expectAttribute('my.module:Customer#name', 'Acme Corp');
});
```

This test survives:
- CSS/layout changes (no selector dependency)
- Field reordering (testid stays the same)
- Component refactoring (rendering internals don't matter)

It only breaks when the **model itself changes** (attribute renamed/removed), which is the correct time to update a test.

### Layer 4: Auto-fix via model diff (future)

Since model changes in TopLogic are structured (XML diffs between versions), a maintenance tool can:

1. Diff the model between two commits/releases
2. Detect renames: `my.module:Customer#name` -> `my.module:Customer#fullName`
3. Mechanically rewrite all test files with the updated references

```bash
# Hypothetical CLI tool
tl-test-migrate --from v7.10 --to v7.11 --test-dir tests/e2e/
# Output: Updated 23 references in 8 test files
```

This is feasible because model references are structured identifiers (not arbitrary CSS selectors), making find-and-replace reliable.

## Implementation Steps

### Step 1: `data-testid` emission (Java, server-side)

- Identify the rendering code path for `FormField` / `FormMember`
- When a field corresponds to a model attribute, emit `data-testid="module:Type#attribute"`
- Ensure this works for: text fields, select boxes, checkboxes, date pickers, table columns
- Add a global config flag to enable/disable testid emission (disable in production for minimal HTML size)

### Step 2: Playwright project setup

- Add a `tests/e2e/` directory (or a dedicated Maven module)
- `package.json` with `@playwright/test` dependency
- `playwright.config.ts` pointing to a running TopLogic instance
- Basic smoke test: login, navigate, verify page title

### Step 3: `TLTestHelper` library

- Implement the helper class as described in Layer 2
- Add convenience methods for common TopLogic patterns (login, navigation, form CRUD)
- Handle TopLogic-specific UI patterns (AJAX updates, dialog confirmations)

### Step 4: Pilot tests

- Write 5-10 tests against `com.top_logic.demo` covering:
  - Login/logout
  - CRUD operations on a demo type
  - Table filtering/sorting
  - Navigation between views
- Validate that tests survive a minor UI refactoring

### Step 5: CI integration

- Add a Playwright test stage to the Jenkins pipeline
- Run against a deployed demo instance
- Report results alongside existing unit tests

### Step 6 (future): Model-diff migration tool

- Parse TopLogic model XML files
- Diff attribute/type references between versions
- Rewrite `data-testid` references in test files

## Open Questions

1. **Where exactly in TopLogic's rendering pipeline should `data-testid` be emitted?** Need to trace the form field rendering from model attribute to HTML output.
2. **Should `data-testid` also cover non-form elements?** (e.g., navigation tabs, toolbar buttons, tree nodes)
3. **How to handle composite fields?** (e.g., a date range with two inputs for the same attribute)
4. **Test data management:** Should tests use a fresh database per run, or seed known data?
5. **Authentication:** How to handle login in tests — reusable auth state via Playwright's `storageState`?
