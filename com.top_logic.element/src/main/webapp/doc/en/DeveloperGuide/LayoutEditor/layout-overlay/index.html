Sometimes it is not sufficient to use only the main layout XML file to determine the layout of a specific application or view. This can be the case, for example, if an application requires a customized display or additional components that are only relevant for this application or a specific module.

How layout overlay works

In a *.layout.overlay.xml file, the parent-child relationship does not play a role. Instead, components of the view to be customized are referenced in this file via their local name. The content of this file represents an incremental configuration that describes changes or additions to existing components.

Use case: Adaptation of the model-based searchUse case: Adaptation of the model-based search

One example is the model-based search, which is defined in the Element module. If you want to implement a different version of this search in an additional module, such as a demo module, for example by adding a component with specific information, a *.layout.overlay.xml file can be created.

Suppose you want to change the model-based search in the demo module by defining additional columns and changing the title key of a component. This could look like this in a file such as com.top_logic.element/search/searchinput.layout.overlay.xml:

<components>
   <component name="SearchAttributedSearchInput" columns="columns(com.top_logic.reporting/search/searchResult.layout.xml#SearchAttributedSearchResultTable)"
   />
   <securityLayout name="searchInput"
      titleKey="pos.search.input.tabber"
   />
</components>
